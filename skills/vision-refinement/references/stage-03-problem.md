# Stage 3 — Problem & Value Definition

## Role

Product strategist defining the problem rigorously.

## Context Required

Read `01-vision-brief.md` and `02-personas.md` before starting.

## Boundaries

- No solution design — focus on the problem space.
- Challenge assumptions about why existing solutions fail.

## Conversation Flow

1. Map current alternatives — what do users do today? List specific tools, workarounds, manual processes.
2. Why do those alternatives fail? Be specific:
   - Too expensive? Too complex? Missing a key feature?
   - Wrong mental model? Bad UX? Poor reliability?
3. Identify friction points in the current experience — where does the workflow break?
4. Define what must absolutely work on day one (non-negotiable functionality).
5. Define what failure looks like — if this product launches and fails, what went wrong?

## Output Artifact: `03-problem-value.md`

```markdown
# Problem Statement & Value Proposition — [Product Name]

## Market Context
[What exists today? How do users currently solve this problem?]

## Core Friction
[Where exactly does the current experience break down?]

## Why Existing Solutions Fall Short
[Specific failures of current alternatives — not vague complaints]

## Value Proposition
[One clear statement: "For [user] who [need], [product] provides [benefit] unlike [alternatives] because [differentiator]."]

## Critical Success Conditions
[What must be true for this product to succeed? List 3-5 concrete conditions.]
```

## Quality Gates

- [ ] Current alternatives are named specifically (not "other apps")
- [ ] Failures of alternatives are evidence-based, not assumed
- [ ] Value proposition follows the template and is defensible
- [ ] Critical success conditions are testable
